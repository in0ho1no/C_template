# CLAUDE.md

## 共通ルール

- 大きな変更の前には短い計画を提示してユーザーの確認を取る
- 次の変更は事前確認を必須とする
  - 依存パッケージの追加
  - 破壊的変更
  - 複数ファイルにまたがる大規模リファクタ
  - 設定ファイルや開発フローの変更
- 変更後は影響範囲に応じて必要最小限の検証を行う

## リポジトリ概要

組み込み C 開発（主に Renesas RX651 系）向けのリポジトリ。
C 言語ガイドライン・規約・組み込み固有ルールを `.github/` 以下で管理している。

## 作業前の確認ファイル

- 共通ルール: `.github/copilot-instructions.md`
- C 言語ガイドライン (`src/**/*.{c,h}` 編集時): `.github/instructions/c.instructions.md`
- 組み込み C ガイドライン (`src/**/*.{c,h,s,S,ld}` 編集時): `.github/instructions/embedded-c.instructions.md`

- C 規約 Markdown 格納先: `.github/rules/c-language/`

## 必須確認事項

### C 言語・処理系

- 言語仕様と処理系依存の挙動を分けて説明する
- 特定コンパイラの拡張構文・最適化・警告挙動・ヘッダ実装を C 言語全体の一般則として扱わない
- mingw など汎用 PC 向け処理系を基準例として持ち込まない
- 処理系依存の話が必要なら、対象コンパイラ名と前提を明記する

### 組み込み開発

- 対象 MCU、ボード、コンパイラ、リンカ、RTOS 有無、起動シーケンスが未確定なら仮定を書いたうえで提案する
- RX 系固有のレジスタ名・内蔵周辺・割り込み制御を一般論として扱わない
- レジスタアクセスは既存の HAL/BSP/CMSIS 相当層を優先し、直接アクセスは必要性が明確な場合に限る
- 割り込み文脈と通常文脈の共有データでは volatile だけで十分か決め打ちせず、排他・原子性・メモリ順序を確認する
- 初期化コードではクロック→ピン→メモリ→周辺→割り込み許可の順序依存を崩さない
- 動的メモリ確保・標準入出力・浮動小数点の使用可否はコスト込みで前提確認してから提案する

### 規約参照

- 命名・コメント・マクロ・型・ヘッダ構成が必要なら `.github/rules/c-language/` 配下の該当 Markdown を先に確認する
- 規約の全文を CLAUDE.md や instructions に重複転記しない
