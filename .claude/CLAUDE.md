# CLAUDE.md

## 共通ルール

- 大きな変更の前には、短い計画を提示してユーザーの確認を取る
- ファイル削除が必要な場合でも、Agentは削除操作を自ら実行しない。削除対象のパスを明示し、削除が必要な旨をユーザーへ通知して判断を仰ぐ
- 次の変更は事前確認を必須とする
  - 依存パッケージの追加
  - 破壊的変更
  - 複数ファイルにまたがる大規模リファクタ
  - 設定ファイルや開発フローの変更
- データ形式や外部フォーマットの処理は、原則として実績ある専用ライブラリを優先する。未導入の場合は依存追加可否を先に確認し、不可の場合のみ低レベルな直接解析などの代替手段を選ぶ
- 検索はVS Code組み込み検索または標準コマンドを優先し、`rg`は導入済みが確認できる場合のみ利用すること
- 変更後は影響範囲に応じて必要最小限の検証を行う
- 根拠のない回答はしない。不明なら不明な旨を伝える。
- 同じ説明を繰り返すのは避ける。

##  Fable5などのフロンティアモデル利用時の注意

### 出力の簡潔さ
- 結論を先に述べる。詳細や理由づけはその後
- 採用しなかった選択肢の長い説明、過剰に構造化されたPR説明、
  自明なコメント（次の行の説明など）は書かない
- 簡潔さは文を断片化することではなく、内容を厳選すること

### 進捗報告の裏付け
- 完了・検証済みと報告する前に、そのセッション内のツール実行結果
  （テスト出力、diff、ビルドログ等）に基づいているか確認する
- 未検証の場合はその旨を明示する。失敗した場合は出力とともに正直に報告する

## 必須確認事項

### C 言語・処理系

- 言語仕様と処理系依存の挙動を分けて説明する
- 特定コンパイラの拡張構文・最適化・警告挙動・ヘッダ実装を C 言語全体の一般則として扱わない
- mingw など汎用 PC 向け処理系を基準例として持ち込まない
- 処理系依存の話が必要なら、対象コンパイラ名と前提を明記する

### 組み込み開発

- 対象 MCU、ボード、コンパイラ、リンカ、RTOS 有無、起動シーケンスが未確定なら仮定を書いたうえで提案する
- RX 系固有のレジスタ名・内蔵周辺・割り込み制御を一般論として扱わない
- レジスタアクセスは既存の HAL/BSP/CMSIS 相当層を優先し、直接アクセスは必要性が明確な場合に限る
- 割り込み文脈と通常文脈の共有データでは volatile だけで十分か決め打ちせず、排他・原子性・メモリ順序を確認する
- 初期化コードではクロック→ピン→メモリ→周辺→割り込み許可の順序依存を崩さない
- 動的メモリ確保・標準入出力・浮動小数点の使用可否はコスト込みで前提確認してから提案する

### 規約参照

- 命名・コメント・マクロ・型・ヘッダ構成が必要なら `.github/rules/c-language/` 配下の該当 Markdown を先に確認する
- 規約の全文を CLAUDE.md や instructions に重複転記しない
